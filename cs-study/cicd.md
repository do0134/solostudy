# CI/CD
<hr/>

## 파이프라인
- 코드구축부터 배포까지 일련의 과정들을 CI/CD 파이프라인이라고 함 
- 3가지 단계로 이루어짐
    1. Continuous Integration : 코드를 빌드하고 테스트하고 합침
    2. Continuous delivery : 해당 레포지토리에 릴리스
    3. Continuous Deployment : 이를 프로덕션 즉, 실제 서비스에 배포
- 장점 : 코드 배포까지 체계적으로 만듦 + 테스트가 강제된다는 점
  - 테스트가 강제되는 것?
    - 머지했을 때 오류나면 머지가 되지 않는다.

### 빌드

### 웹팩
> 여러 가지 모듈들을 정적 자산으로 빌드해주는 도구
> 즉, 코드를 사용자가 볼 수 있는 자산(html, css, javascript 등)로 바꿔준다

### 테스트
> 함수 등 작은 단위를 테스팅하는 단위테스트, 모듈을 통합할 때 테스트하는 통합테스트, 사용자가 서비스를 사용하는 상황을 가정해서 테스트하는 엔드투엔드 테스트가 대표적
> 대표적인 프레임워크론 mocha가 있음

### 머지 
- git이나 svn을 통해 머지
- 충돌을 방지하기 위해서 작은 프로젝트는 특정 디렉토리를 해당 개발자만 맡는다.
- 그런데 충돌은 어차피 일어남. 그래서 작은 단위로 충돌이 일어나게 하는 게 중요!
- 작은 이슈 단위로 보통 머지함.
- 충돌 시에는 화면공유하며 합의하에 충돌을 해결하는게 제일 좋다!

### 배포
- 사용자를 위한 서비스를 배포할 뿐은 아니다!
- QA 엔지니어나 Admin 페이지를 위한 배포, 데이터웨어하우스부터 데이터를 가공해서 백엔드 개발자를 위한 배포 등 포함

### 툴
- github action, genkins, circle ci가 유명하며, heroku를 통해 CI, CD 설정없이 자동 가능 
- 참고로 heroku + github action으로 설정도 가능
